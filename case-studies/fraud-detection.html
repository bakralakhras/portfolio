Real-Time Fraud Detection Platform

A unified streaming + batch fraud analytics platform built on a Kafka-to-Iceberg lakehouse — one storage layer, two access paths, no duplicated data.

Overview

Fraud teams need two things from data: real-time alerts when a suspicious pattern fires, and historical analysis to tune the rules that produced those alerts. Most teams build these as separate pipelines on separate storage. We built one.

The Problem

The legacy setup had a Kafka → custom alerting service for the real-time path, and a separate nightly ETL into a relational warehouse for analysts. Two failure modes followed: analysts and ML engineers worked on different data than the rules engine, and storage costs were paid twice for the same events.

The Approach

Treat Kafka as the ingestion bus and Iceberg-on-MinIO as the single source of truth. Spark Structured Streaming powers the real-time path; the same Iceberg tables are queried by Trino for ad-hoc analytics and Superset dashboards.

Key Decisions & Tradeoffs

Iceberg over Delta Lake

Decision
Chosen
Rejected
Table format
Apache Iceberg
First-class engine support across Spark, Trino, Flink. Rich snapshot model. Catalog-agnostic.
Delta Lake
Strongest in the Spark/Databricks lane; weaker support in Trino at the time of evaluation.
Why it matters
We needed equal first-class read performance from Trino for analysts and Spark for the streaming engine. Iceberg won on engine parity.
Open table format claims aside, the Trino integration story decided this.

Spark over a separate Flink cluster

Flink is the better tool for sub-second streaming with rich state. We dropped it anyway. Operating a second cluster type — a second set of dashboards, runbooks, on-call patterns — wasn't justified by the latency win at our SLO of p95 < 5s. Spark Structured Streaming hit it; the team already knew Spark.

Operational simplicity over latency wins

If our SLO required < 1s, Flink would be back on the table. Choose by the SLO, not by the benchmark.

Implementation Notes

Streaming sink with exactly-once semantics

jobs/transactions_to_iceberg.py python 
# Read from Kafka, evaluate rules, sink raw + alerts to Iceberg
df = (spark.readStream
        .format("kafka")
        .option("subscribe", "transactions")
        .option("startingOffsets", "latest")
        .load())

events = parse_avro(df.value, schema_registry)

(events.writeStream
    .format("iceberg")
    .outputMode("append")
    .option("checkpointLocation", "s3a://chk/transactions/")
    .trigger(processingTime="30 seconds")
    .toTable("lake.fraud.transactions"))

alerts = apply_rules(events)
alerts.writeStream.format("kafka") \
    .option("topic", "alerts.fraud") \
    .option("checkpointLocation", "s3a://chk/alerts/") \
    .start()

Schema evolution without rewrites

Producers send Avro with Confluent Schema Registry. Iceberg's column-id-based schema means adding nullable fields is a metadata operation; old files are read with default values. We added 12 schema changes in the first six months without rewriting a single Parquet file.

Outcomes

  • Single storage layer for streaming and analytics — analysts and the rules engine read the same tables, eliminating the "real-time vs. batch numbers don't match" class of bugs.
  • Schema evolution at metadata speed — additive schema changes are zero-downtime and don't require Parquet rewrites.
  • Replayable — Iceberg snapshots + Kafka retention let us reprocess historical windows against new rules without breaking live ingestion.

What I'd Do Differently

Compaction was an afterthought. Streaming sinks produce many small files; query performance degraded measurably by month two. Plan compaction from day one — Airflow DAGs running rewrite_data_files on a schedule, with snapshot expiration to bound metadata growth.
