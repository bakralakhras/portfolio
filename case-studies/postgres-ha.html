HA PostgreSQL with Patroni

A self-managed, regulator-compliant PostgreSQL cluster that fails over automatically — built on Patroni, etcd, HAProxy, and Keepalived on bare RHEL.

Overview

The workload was regulated. Data residency requirements ruled out managed PostgreSQL services. The team needed automatic failover, point-in-time recovery, and an operational story that didn't depend on heroics during a 3 AM page.

The Problem

Single-node PostgreSQL with cron-based pg_dump backups is fine — until the day it isn't. The previous setup had no failover, recovery was a manual restore from yesterday's dump, and there was no procedure that survived contact with a real outage.

The Approach

Three RHEL nodes running Patroni-managed PostgreSQL, with etcd providing the consensus layer. HAProxy in front for connection routing; Keepalived for VIP failover at L4. WAL archived continuously to MinIO via pgBackRest, with weekly point-in-time recovery drills validating the backups are usable, not just present.

Key Decisions & Tradeoffs

Patroni over repmgr

Decision
Chosen
Rejected
Cluster manager
Patroni + etcd
Declarative cluster topology in YAML. Leader election delegated to a real consensus store.
repmgr
Lighter, but failover is more script-ish. Less battle-tested in failure scenarios involving partitioned witnesses.
Why it matters
When the leader is sick but not dead, Patroni's behavior is predictable because the truth lives in etcd, not in scripts running on the sick node.
repmgr would have worked. Patroni's REST API made HAProxy integration cleaner, which sealed it.

etcd over Consul

The platform already ran etcd for other components. Adding Consul would have been a second consensus store with its own ops profile. One quorum to operate beat any feature edge Consul offered for this workload.

HAProxy + Keepalived over PgBouncer-only

Layer
Tool
Why
L4 failover (VIP)
Keepalived
Clients reconnect to a stable address; no DNS TTL games during failover.
Leader-aware routing
HAProxy
Health-checks Patroni's REST endpoint. Routes writes to current leader, reads to replicas.
Pooling
PgBouncer (per app)
Application-side pool to dampen connection storms. Not the failover layer.
Backups exist if and only if you've restored them

Weekly automated PITR drills against an ephemeral instance. If the drill fails, the cluster is paged — backups are a tier-1 service, not a cron job.

Implementation Notes

HAProxy backend with Patroni health checks

/etc/haproxy/haproxy.cfg haproxy 
listen postgres-write
    bind *:5432
    option httpchk OPTIONS /leader
    http-check expect status 200
    default-server inter 3s fall 3 rise 2 on-marked-down shutdown-sessions
    server pg-01 10.0.0.11:5432 check port 8008
    server pg-02 10.0.0.12:5432 check port 8008
    server pg-03 10.0.0.13:5432 check port 8008

listen postgres-read
    bind *:5433
    option httpchk OPTIONS /replica
    http-check expect status 200
    balance roundrobin
    server pg-01 10.0.0.11:5432 check port 8008
    server pg-02 10.0.0.12:5432 check port 8008
    server pg-03 10.0.0.13:5432 check port 8008

Patroni exposes /leader returning 200 only when the node is the elected leader, and /replica returning 200 only on healthy replicas. HAProxy uses these as the source of truth — no health-check logic lives in scripts.

Outcomes

  • Automatic failover in ~12 seconds — application sees a brief connection blip, no human in the loop required.
  • RPO under 1 minute via continuous WAL archiving to MinIO with pgBackRest.
  • Tested backups — weekly automated PITR drills mean a restore is a known-good procedure, not a hope.
  • Read scaling — replicas serve read-only workloads through the dedicated HAProxy port, taking pressure off the leader.

What I'd Do Differently

Spend more time on the application reconnect story earlier. Half of the early "failover incidents" weren't database problems at all — they were application connection pools holding stale connections. Document the recommended client-side timeout and pool-recycle settings, and ship them as defaults to internal services from day one.
