Platform Engineering Production

Sovereign Data Platform

A GitOps-managed data lakehouse on OpenShift 4.18 — 10 stateful services, zero static credentials, a single bootstrap command. Built to be operated declaratively with zero configuration drift.

Overview

The platform needed to host streaming, batch, and analytical workloads on private infrastructure — no managed cloud, no escape hatch. Every cross-cutting concern had to be solved in-cluster: secrets lifecycle, certificate rotation, identity, backups, long-term metrics. The constraint that shaped every decision was operational simplicity at scale: a two-engineer on-call rotation had to be able to run this on a Friday night.

The Problem

The existing approach — Helm umbrella charts covering the whole platform — had two failure modes we kept hitting in early rollouts. First, a Trino version bump couldn't ship without staging the entire platform. Second, the moment any engineer touched a cluster directly, we'd accumulate configuration drift that was invisible until something broke.

The harder constraint was the infrastructure boundary. This platform sits on private on-premises infrastructure. There are no managed equivalents for secrets management, certificate rotation, or object storage. Every capability the platform needed had to be built and maintained in-cluster, by the same team operating it.

I decided early that GitOps wasn't a nice-to-have. It was the only operating model that could survive a two-engineer on-call rotation across 10 stateful services. Everything else followed from that.

The Approach

ArgoCD with the app-of-apps pattern became the control plane. Each platform component gets its own ArgoCD Application with its own promotion cadence. A parent root app composes them. Vault handles the secret layer — nothing in Git references a credential except by its Vault path. Adding a new component means one new file in apps/ and one new folder in platform/. Nothing else.

Key Decisions & Tradeoffs

App-of-apps over Helm umbrella

Decision
Chosen
Rejected
Composition model
ArgoCD app-of-apps
Each component is its own Application, sourced from its own Git path. The root app composes them. Sync waves enforce startup order deterministically.
Helm umbrella chart
One release for the entire platform. A single bad values.yaml merge re-renders every sub-chart simultaneously. Blast radius is invisible until it isn't.
Why it matters
Trino ships on its own schedule without staging Kafka. A cold restore boots in wave order: Vault, cert-manager, storage, data plane, observability. Reproducible from Git alone.
Umbrella charts collapse promotion cadence across unrelated services. Debugging requires unwrapping nested Helm values that don't map cleanly to what's running.

Vault Secrets Operator over static secrets

Decision
Chosen
Rejected
Secrets lifecycle
VSO + Vault KV v2 (AppRole)
VSO reconciles secrets every 30 seconds from Vault paths. AppRole gives explicit, auditable role-based access. Zero hardcoded credentials anywhere in Git or etcd.
Static Kubernetes Secrets
Checked into Git, intentionally or accidentally, visible in kubectl get secret, no TTL, no audit trail, no rotation story.
Why it matters
Rotation is a Vault policy change, not a deployment. Compromised credentials have a hard 30-second blast radius before VSO overwrites them.
We chose AppRole over Kubernetes auth specifically because VSO 1.3.0 dropped token auth. AppRole is explicit; there's no ambient service account identity to misuse.
Trade-off accepted

VSO's internal client cache doesn't invalidate when you update the AppRole secret-id in Kubernetes. Discovered this the hard way: updating the secret requires oc rollout restart of the VSO controller manager. Documented in the Phase 7 runbook.

Thanos Querier over standalone Prometheus

Rather than deploying a parallel Prometheus stack, I piggybacked on OpenShift's built-in User Workload Monitoring. A custom Grafana instance, managed by Grafana Operator v5, points at the Thanos Querier endpoint at thanos-querier.openshift-monitoring.svc:9091, authenticated via a long-lived ServiceAccount token with cluster-monitoring-view. This gave us cross-namespace metric federation without the operational overhead of managing a second Prometheus lifecycle.

Real Engineering Problems

These aren't lessons retrofitted after the fact. These are the actual problems that blocked progress and required root cause analysis to resolve.

The ArgoCD CRD chicken-and-egg

Incident

ArgoCD sync waves don't solve missing CRDs because ArgoCD validates manifests before it applies them. Deploying the Vault Secrets Operator subscription and the VSO CRD-dependent resources in the same sync wave caused validation failures on every sync. The fix was splitting them into two separate ArgoCD Applications: one for the Subscription, which creates the CRDs, and one for the CRD-dependent VaultConnection and VaultAuth resources. Wave ordering across Applications works where wave ordering within a single Application doesn't.

The WaitForFirstConsumer deadlock

Incident

The Helm-installed StorageClass shipped with volumeBindingMode: WaitForFirstConsumer. This is incompatible with local storage: local PVs bind immediately or not at all. The catch is that volumeBindingMode is immutable on a StorageClass. You can't patch it. Full StorageClass deletion and recreation with Immediate binding was the only path forward. Any PVCs already bound to the old class had to be migrated.

NiFi 2.x SNI rejection on OpenShift Routes

Incident

NiFi 2.x runs on Jetty 10, which enforces strict SNI validation against its auto-generated keystore. The keystore has CN=localhost. OpenShift Routes send the actual hostname. SNI mismatch, connection refused. Jetty 9, used by NiFi 1.27.0, doesn't enforce SNI at this level. Downgraded to 1.27.0. NiFi 2.x on OpenShift Routes requires a custom keystore with the route hostname as a SAN.

Superset 5.0.0 hardened image has no pip in venv

Incident

The Superset 5.0.0 hardened image strips pip from its virtualenv. The standard Trino driver install pattern, pip install trino in an init container, fails silently: the package installs but doesn't land where Superset can find it. Workaround: pip install --target=/tmp/pkgs --no-deps combined with PYTHONPATH=/tmp/pkgs. The --no-deps flag is critical because SQLAlchemy 2.0 conflicts with the venv's pinned-compatible version.

OpenShift service env injection collision

Incident

OpenShift automatically injects environment variables for every service in the namespace. For a service named superset, it injects SUPERSET_PORT=tcp://10.96.x.x:8088. Superset's run-server.sh expects SUPERSET_PORT to be a numeric port. The string tcp://... breaks the startup script with a non-obvious error. Fix: explicitly override SUPERSET_PORT=8088 in the deployment manifest.

Implementation: The Bootstrap

The entire platform deploys from a single command after ArgoCD is installed. The root application points at apps/ in the Git repository. Every YAML file in that directory is an ArgoCD Application for one platform component. ArgoCD discovers and reconciles them automatically.

apps/root.yaml yaml 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: platform-root
  namespace: openshift-gitops
spec:
  project: platform
  source:
    repoURL: https://github.com/bakralakhras/OpenShift-ArgoCD.git
    path: apps/           # every yaml here becomes a component app
    targetRevision: main
  destination:
    server: https://kubernetes.default.svc
    namespace: openshift-gitops
  syncPolicy:
    automated: { prune: true, selfHeal: true }
    syncOptions: [CreateNamespace=true, ApplyOutOfSyncOnly=true]

Sync waves handle cold-start ordering. Vault must be healthy before cert-manager runs. Storage must be ready before data services boot. A failed wave blocks everything downstream, which is exactly the right behavior for a platform with hard dependency ordering.

Outcomes

  • Zero static credentials across the entire platform. Every secret is materialized by VSO from a Vault KV v2 path, reconciled every 30 seconds.
  • Independent component release cadence. Trino, Kafka, and Airflow each ship on their own schedule. The root app composes the platform; it doesn't gate it.
  • Cold restore from Git alone. A fresh OpenShift cluster plus one oc apply reproduces the entire platform in wave order. No tribal knowledge required.
  • Two-engineer on-call. Per-phase runbooks cover every component. Sync-wave bootstrap makes a cold restore predictable and sequenced.
  • Nine production incidents documented. Every failure captured with symptom, root cause, and fix, available in the phase runbooks in the repository.

What I'd Do Differently

The policy layer should have been day one. I rolled Kyverno retroactively to enforce things like "no hardcoded imagePullSecrets" and "every namespace requires a NetworkPolicy." Three production-bound misconfigurations during early rollouts would have been caught at admission time if the guardrails were there from the start. Kyverno is cheap to run; the cost of retrofitting policies onto a running platform is not.

The other miss was treating Vault as infrastructure rather than a service. Early on, a Raft leader election storm masked itself as a Kafka client issue for nearly an hour. We now alert on Vault leader transitions, seal status, and policy denial rate as first-class signals, the same way we alert on Kafka consumer lag or Trino query failures. If your platform boots through Vault, Vault needs to be in your top-five observability concerns from day one.
