AI Incident Responder

An async pipeline that consumes Alertmanager webhooks, enriches them with Prometheus context, and produces structured remediation suggestions via an LLM — without becoming yet another flaky chatbot.

Overview

Multi-cluster on-call has a familiar pattern: an alert fires, the responder spends the first three minutes just figuring out where to look. This project closes that gap — every alert gets enriched with the metric context, the recent deploys, and a structured suggestion of what to check first.

The Problem

Alertmanager fires an alert. The pager goes off. The responder logs into Grafana, picks the right dashboard, scrolls, finds the relevant panel, then opens the runbook. Most of that is mechanical. The interesting work — deciding what to actually do — comes after.

The Approach

Treat the LLM as a structured output generator, not a chatbot. Webhook in → enrich with measurable context → render a strict prompt template → demand JSON back → render that JSON into a Slack message. No freeform reasoning surfaced to humans on-call.

Key Decisions & Tradeoffs

Async Celery queue over synchronous API

Decision
Chosen
Rejected
Webhook handling
Webhook → queue → worker
Webhook returns 200 immediately. Workers consume, retry, dead-letter.
Inline LLM call
Webhook holds the connection through a 4–8s LLM call. Storms cause cascading timeouts.
Why it matters
Alert storms are exactly when this thing must not crater. Decoupling makes burst capacity a queue-depth knob.
Alertmanager will retry on 5xx — synchronous failures duplicate work.

Structured prompts over freeform

The prompt is a Jinja template that produces a request demanding a strict JSON schema: likely_cause, checks (an array of three steps), severity_assessment, related_runbook. The response is parsed against a Pydantic model; anything that fails validation gets a single retry with the validation error appended, then falls back to a "context-only" message.

The LLM never sees the responder

Output is rendered, not relayed. If the JSON is malformed, the human still gets the enriched context — the LLM piece degrades gracefully.

FastAPI over Flask

Native async support matters when the worker side is doing concurrent Prometheus queries. Pydantic for the request schema is a free win — Alertmanager's webhook payload becomes a typed object at the door.

Implementation Notes

The validator loop

workers/triage.py python 
class Triage(BaseModel):
    likely_cause: str
    checks: list[str] = Field(min_length=3, max_length=3)
    severity: Literal["low", "medium", "high"]
    related_runbook: str | None

async def triage(alert: Alert) -> Triage | None:
    ctx = await enrich(alert)            # prom + deploys + runbooks
    prompt = render("triage.j2", alert=alert, ctx=ctx)

    for attempt in range(2):
        raw = await llm.complete(prompt, response_format="json")
        try:
            return Triage.model_validate_json(raw)
        except ValidationError as e:
            prompt = retry_prompt(prompt, raw, e)

    return None  # caller renders context-only fallback

Outcomes

  • Bounded LLM behavior — strict JSON schema means the system either produces a usable triage object or no triage at all. No half-rendered hallucinations on a pager.
  • Survives alert storms — async queue absorbs bursts; workers scale horizontally; webhook stays cheap.
  • Graceful degradation — when the LLM is unreachable or returns garbage, the on-call still gets an enriched message with the metric snapshot and recent deploys.

What I'd Do Differently

Add an evaluation harness from week one. The system is only as good as its prompt; without recorded ground-truth alerts and a way to diff prompt changes against them, prompt iteration is vibes. The next iteration replays a corpus of historical alerts on every prompt change.
