Observability Engineering Completed

Distributed NiFi Monitoring Stack

Full-stack observability across 20 geographically distributed sites: 60 NiFi nodes, 120 scrape targets, no single point of failure, and one URL for the client. Every architectural decision was forced by a constraint the previous one exposed.

Overview

The requirement was straightforward on paper: build observability for a distributed NiFi deployment spanning 20 sites across multiple cities, each running a 3-node NiFi cluster. Two things had to be monitored per node: the NiFi flow metrics via the API on port 8443, and the underlying server via node exporter. The client wanted a single dashboard URL, with no infrastructure complexity exposed to them.

What looked like a monitoring project became a storage architecture problem, a vendor licensing problem, and an HA design problem, in that order. Each decision revealed the next constraint.

The Decision Chain

This stack wasn't designed top-down. It was built decision by decision, each one exposing the next problem. This is the actual sequence.

1
Decision
Prometheus to scrape NiFi API and node exporter
The target was clear: scrape NiFi's API metrics on port 8443 for processor stats, connection queue depth, flow throughput, and node exporter on each VM for server-level visibility. Prometheus was the obvious starting point.
2
Constraint
One Prometheus instance is a single point of failure
120 scrape targets across 20 cities through a single Prometheus instance was operationally unacceptable. If that instance went down, monitoring for the entire deployment went blind. Two Prometheus instances were needed for HA.
3
Decision
Two Prometheus instances in agent mode
Both scrape the same 120 targets and forward metrics upstream via remote write. Agent mode removes local storage from Prometheus entirely, turning it into a stateless scrape-and-forward process instead of a stateful database.
4
Constraint
Two Prometheus instances produce duplicate metrics
Both agents scrape the same targets and forward identical time series. Without deduplication, every downstream query appears twice, dashboards show doubled values, alerts fire twice, and aggregations are wrong.
5
Decision
Thanos for deduplication and long-term storage
Thanos Receive ingests the remote write streams from both Prometheus agents. Thanos Querier deduplicates at query time using the replica label. Two Thanos Receive instances provide HA, with Keepalived exposing a VIP for the write endpoint.
6
Constraint
Thanos needs object storage, and MinIO was the default assumption
Thanos ships blocks to an S3-compatible object store for long-term retention. The available storage infrastructure was NFS. MinIO and NFS are incompatible because MinIO requires direct I/O semantics that NFS cannot reliably provide.
7
Constraint
MinIO Community Edition was a long-term fit risk
Even setting aside the NFS incompatibility, the client required fully open-source software. MinIO Enterprise was not an option, and relying on CE was not acceptable for a production deployment under those constraints.
8
Resolved
SeaweedFS as the object store backend
SeaweedFS is S3-compatible, open-source, and designed to work on top of NFS storage. Two SeaweedFS nodes running on two NFS storage servers provide HA at the object store layer. Thanos treats it as a standard S3 endpoint.
9
Resolved
Keepalived VIP in front of Grafana for a single client endpoint
Two Grafana instances run active/passive behind a Keepalived VIP. The client gets one URL. If the active Grafana node fails, Keepalived promotes the standby and the VIP moves. The same pattern fronts Thanos Receive for the Prometheus agent write endpoint.

Key Decisions & Tradeoffs

Prometheus agent mode over standard Prometheus

Decision
Chosen
Rejected
Scraping model
Prometheus agent mode
Stateless scrape-and-forward. No local TSDB. Both agents scrape all 120 targets and remote_write to the Thanos VIP. Deduplication happens at query time in Thanos Querier.
Standard Prometheus + load balancer
Requires managing local storage per Prometheus instance. A load balancer can split scrape targets unpredictably, so partial failure creates partial blind spots with no clear ownership.
Why it matters
Agent mode makes Prometheus operationally simple. No disk management, TSDB compaction, or retention tuning. If an agent dies, the other still has complete coverage.
A load balancer approach solves one HA problem but creates a split coverage problem. Agent mode with full duplication avoids this entirely.

Keepalived VIP over load balancer

Decision
Chosen
Rejected
HA mechanism
Keepalived + VIP
A virtual IP floats between active and standby nodes. Used in front of both Thanos Receive for Prometheus writes and Grafana for client access. No external dependency.
External load balancer
Introduces another infrastructure component to manage and monitor. In an environment where the client controls the hardware and wants minimal moving parts, this was overhead without proportional benefit.
Why it matters
Keepalived gives deterministic failover: the VIP moves to the standby within seconds of a VRRP heartbeat failure. For Grafana, the client never learns there are two nodes.
At two-node HA, a load balancer adds latency, complexity, and a potential single point of failure unless it is also made redundant.

SeaweedFS over MinIO

Decision
Chosen
Rejected
Object store
SeaweedFS on NFS
S3-compatible, fully open-source, and designed to run on top of NFS. Two nodes for HA. Thanos configuration unchanged because it sees a standard S3 endpoint.
MinIO CE
Incompatible with NFS storage backends. CE roadmap created a long-term fit risk, while Enterprise conflicted with the open-source requirement.
Why it matters
SeaweedFS was the S3-compatible open-source solution that fit the available NFS storage layer and kept the object store abstraction clean for Thanos.
MinIO on NFS can corrupt data under concurrent writes. This is a fundamental incompatibility between MinIO's I/O model and NFS semantics, not a tuning issue.

Scraping NiFi

Each NiFi node exposes two metric sources. The NiFi API on port 8443 provides flow-level observability: processor execution counts, connection queue depth and byte count, back-pressure events, and bulletin board errors. Node exporter on each VM provides the server layer: CPU, memory, disk I/O, and network throughput.

The NiFi API requires authentication. Prometheus scrapes it with a bearer token configured per job in the scrape config. The TLS endpoint uses NiFi's internal keystore, so the Prometheus job initially used insecure_skip_verify: true, with CA validation tracked as a certificate-hardening follow-up.

prometheus-agent.yml yaml 
scrape_configs:
  - job_name: nifi-api
    scheme: https
    tls_config:
      insecure_skip_verify: true   # replace with ca_file during cert hardening
    authorization:
      credentials_file: /etc/prometheus/nifi-token
    static_configs:
      - targets:
          # site-01
          - site01-nifi-1:8443
          - site01-nifi-2:8443
          - site01-nifi-3:8443
          # ... 17 more sites
    relabel_configs:
      - source_labels: [__address__]
        regex: (site\d+)-.*
        target_label: site
        replacement: ${1}

  - job_name: node-exporter
    static_configs:
      - targets:
          - site01-nifi-1:9100
          - site01-nifi-2:9100
          - site01-nifi-3:9100
          # ... 17 more sites

remote_write:
  - url: http://<thanos-vip>:19291/api/v1/receive
    headers:
      THANOS-TENANT: nifi-platform

The site label injected via relabeling powers the per-site Grafana dashboards. Every metric carries the site it came from, so a single dashboard template can filter by site using a variable.

Deduplication label

Both Prometheus agents add a replica label, such as prom-1 or prom-2, before remote_write. Thanos Querier is configured with --query.replica-label=replica. At query time, Thanos picks one replica and discards the other.

Grafana Dashboards

There are two dashboard layers. Site dashboards give operators a geographic overview: which sites are healthy, which NiFi nodes have elevated queue depth or back-pressure, and which servers are under resource pressure. The site variable is a Grafana template variable populated from the site label.

Flow dashboards drill into a specific NiFi cluster: individual processor throughput, connection queue depth over time, processor errors, and bulletin events. These are the dashboards an operator uses when a site-level alert fires and they need to identify which flow or processor is the cause.

What I'd Do Differently

The insecure_skip_verify on NiFi TLS should have been resolved earlier in the build. The NiFi internal keystore CN doesn't match the hostnames Prometheus scrapes, so proper CA validation requires either importing NiFi's self-signed CA into Prometheus or replacing NiFi's keystore with certs signed by a trusted CA. The gap is documented as a certificate-hardening item.

The site-label relabeling approach scales to 20 sites cleanly, but if this deployment grows significantly, static scrape configs become a maintenance burden. The long-term answer is service discovery: either file-based SD populated by configuration management or DNS-based SD if the site hostnames follow a stable pattern. For 20 sites, static configs are fine. For 50+, they become a problem.
